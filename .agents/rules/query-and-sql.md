# Query And SQL Rules

适用于 `internal/service/` 中涉及复杂查询、聚合查询、分页查询与 ORM / SQL 取舍的场景。

## MUST

- 默认优先使用 GORM / ORM 组合已有实体关系完成查询
- 只有在 ORM 明显无法清晰表达，或已确认存在严重性能问题时，才引入手写 SQL
- 若必须使用 SQL，范围要尽量小，只承载 ORM 难以表达的那一小段逻辑
- 查询实现必须让业务语义优先可读，不得把核心规则埋进大块 SQL 字符串里
- 复杂查询改动后必须检查 PostgreSQL、MySQL 兼容性，不得只按单一数据库习惯书写

## SHOULD

- 优先拆成“多次简单查询 + Go 侧聚合”，而不是一次性拼接超长 SQL
- 优先复用现有分页结构、游标编码与排序约定
- 查询逻辑应按“数据加载、聚合转换、排序分页”分层，避免揉在一个函数里
- 需要兼容数据库差异时，把兼容处理封装在小函数中，不要散落在主流程
- 当查询开始承载独立读模型语义时，可考虑拆到单独的 query / repository 文件

## AVOID

- 不要为图省事直接写 `UNION ALL`、窗口函数、CTE 叠加的大块 SQL，除非确有必要
- 不要把业务规则、分页规则、数据库兼容逻辑全部塞进一个 SQL 或一个函数
- 不要在没有性能证据时为了“看起来更快”提前下沉到原生 SQL
- 不要让 handler 依赖 SQL 细节或数据库字段拼装逻辑

## 取舍原则

- 优先级顺序：可维护性 > 业务可读性 > 迁移成本 > 未证实的性能担忧
- 如果 ORM 方案稍微多查几次但更容易理解，优先接受这种实现
- 如果 SQL 方案明显更短，但业务理解更困难，优先选择更清晰的实现
- 只有在压测、Explain、线上观测等证据表明 ORM 方案成为瓶颈时，才扩大 SQL 比例

## 变更检查

- 这段查询是否还能被不熟悉上下文的同学快速读懂
- 是否可以拆成多个 ORM 查询后在 Go 中聚合
- 是否把分页、排序、聚合、兼容性处理解耦了
- 是否引入了数据库方言相关写法
- 是否补了匹配的 service test
