# Backend Handler Rules

适用于 `internal/handler/` 下的接口、路由和中间件改动。

## MUST

- handler 只负责参数绑定、权限检查、调用 service、组装响应
- 路由统一在 `internal/handler/handler.go` 注册
- 管理接口统一挂到 `/api/v1/admin/*`，并使用管理员鉴权
- 新增接口时优先沿用现有 `/api/v1` 路由结构和命名风格
- 新接口优先复用统一接口设计，不继续扩散已废弃接口模式

## SHOULD

- 请求和响应结构体命名清晰，和 handler 职责对应
- 参数校验放在 handler 入口附近，业务规则校验放到 service
- 错误直接向上传递，由现有错误处理链路决定返回形式
- 分页、游标、过滤条件优先沿用现有字段名与返回格式

## AVOID

- 不要在 handler 中堆业务逻辑、事务、复杂查询或跨实体更新
- 不要在多个文件里重复注册路由
- 不要为了单个接口改动打破已有路由分组和鉴权结构

## 变更检查

- 新接口是否放在正确的路由组下
- 是否需要同步更新前端 API 客户端与类型定义
- 是否误用了已废弃接口风格
- 是否补了必要的 handler 或 service 层验证
